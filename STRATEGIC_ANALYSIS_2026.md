```python
# backend/tasks/sync_tasks.py
from celery_app import celery

@celery.task(bind=True)
def sync_connector_task(self, connector_id, tenant_id):
    """Background task for syncing connector data"""

    # Update status
    self.update_state(state='PROGRESS', meta={'progress': 0, 'status': 'Starting sync...'})

    connector = Connector.query.get(connector_id)
    service = get_connector_service(connector.type)

    # Sync with progress callbacks
    def on_progress(percent, message):
        self.update_state(state='PROGRESS', meta={'progress': percent, 'status': message})

    result = service.sync(connector, progress_callback=on_progress)

    return {'status': 'completed', 'documents': result['document_count']}
```

```python
@integration_bp.route('/integrations/jobs/<job_id>', methods=['GET'])
@jwt_required()
def get_job_status(job_id):
    task = celery.AsyncResult(job_id)

    if task.state == 'PENDING':
        response = {'state': task.state, 'status': 'Pending...'}
    elif task.state == 'PROGRESS':
        response = {
            'state': task.state,
            'progress': task.info.get('progress', 0),
            'status': task.info.get('status', '')
        }
    elif task.state == 'SUCCESS':
        response = {'state': task.state, 'result': task.info}
    else:
        response = {'state': task.state, 'status': str(task.info)}

    return jsonify(response)
```

## 6. APPROACH DEBATES & TRADE-OFFS {#debates}

### Debate #1: Build vs Buy (In-Workflow Delivery)

**Option A: Build Custom Slack/Teams Bots**

**Pros**:
- Full control over UX
- Deep integration with 2nd Brain features
- Custom branding
- No revenue sharing

**Cons**:
- 8-12 weeks engineering effort
- Maintenance burden (Slack/Teams API changes)
- Slack app store approval process

**Cost**: 8 weeks engineering = ~$20K

---

**Option B: Use Embedded Widget Platforms (Intercom, Drift)**

**Pros**:
- 1-2 week implementation
- Proven UX patterns
- Built-in analytics

**Cons**:
- Monthly cost ($50-200/mo)
- Less customization
- Vendor lock-in

**Cost**: $100/mo + 2 weeks engineering = ~$5K setup + $1.2K/year

---

**Option C: Hybrid (Slack Bot + Generic Embed)**

**Pros**:
- Best of both worlds
- Prioritize highest-ROI channel (Slack)
- Use generic embed for others

**Cons**:
- Slightly more complexity

---

**RECOMMENDATION**: **Option C - Hybrid**

**Reasoning**:
- Slack is highest priority (most requested by customers)
- Custom bot provides competitive differentiation
- Generic embed covers other platforms quickly
- Total cost: 4 weeks engineering + $1.2K/year

---

### Debate #2: Knowledge Graph Approach

**Option A: Build Custom Graph Database (Neo4j)**

**Pros**:
- Optimal for graph queries (Cypher)
- Native relationship traversal
- Best performance for complex queries

**Cons**:
- New infrastructure dependency
- Learning curve (Cypher query language)
- Data sync complexity (Postgres ‚Üí Neo4j)

**Cost**: Neo4j AuraDB = $65-500/mo + 6 weeks engineering

---

**Option B: Leverage Existing Pinecone + Postgres**

**Pros**:
- No new infrastructure
- Relationships stored as JSON in Postgres
- Vector similarity for "similar entities"
- Lower cost

**Cons**:
- Limited graph query capabilities
- Slower for deep traversals

**Cost**: $0 additional + 2 weeks engineering

---

**Option C: Hybrid (Postgres for storage, Neo4j for visualization)**

**Pros**:
- Best performance for graph queries
- Postgres remains source of truth
- Sync only when generating visualization

**Cons**:
- Added complexity

---

**RECOMMENDATION**: **Option B - Leverage Existing**

**Reasoning**:
- Graph queries are infrequent (mostly for visualization)
- Pinecone similarity can approximate "related entities"
- Avoid infrastructure complexity
- Can migrate to Neo4j later if needed

---

### Debate #3: Mobile Strategy

**Option A: Native Apps (Swift + Kotlin)**

**Pros**:
- Best performance
- Platform-specific UX
- Full API access

**Cons**:
- 2x development effort (separate codebases)
- 2x maintenance burden
- Requires iOS + Android expertise

**Cost**: 24 weeks engineering = ~$60K

---

**Option B: React Native (Cross-Platform)**

**Pros**:
- Single codebase (80% code shared)
- Faster development (12 weeks vs 24)
- Existing React expertise

**Cons**:
- Slightly worse performance
- Some platform differences

**Cost**: 12 weeks engineering = ~$30K

---

**Option C: Progressive Web App (PWA)**

**Pros**:
- 2 weeks engineering effort
- Works on all platforms
- No app store approvals

**Cons**:
- Limited offline capabilities
- No push notifications (iOS restriction)
- Worse UX vs native

**Cost**: 2 weeks engineering = ~$5K

---

**RECOMMENDATION**: **Option B - React Native**

**Reasoning**:
- Best cost/benefit ratio
- Faster time-to-market vs native
- Better UX vs PWA
- Push notifications work (critical for engagement)
- Can optimize specific screens natively if needed

---

### Debate #4: Proactive Surfacing ML Model

**Option A: Collaborative Filtering (User Behavior)**

**Approach**: "Users like you also read..."

**Pros**:
- Proven approach (Netflix, Amazon)
- Works with limited data
- Good for discovery

**Cons**:
- Cold start problem (new users)
- Privacy concerns (tracking behavior)
- Requires significant user data

**Accuracy**: 60-70%

---

**Option B: Content-Based (Document Similarity)**

**Approach**: "Similar to what you're reading..."

**Pros**:
- No cold start problem
- Works immediately
- No privacy concerns (no behavior tracking)

**Cons**:
- Less serendipitous discovery
- Requires good document metadata

**Accuracy**: 70-80%

---

**Option C: Hybrid (Collaborative + Content)**

**Approach**: Weighted combination

**Pros**:
- Best of both worlds
- Higher accuracy

**Cons**:
- More complex

**Accuracy**: 75-85%

---

**Option D: LLM-Based (GPT-4o Context Reasoning)**

**Approach**: "Given user role and current context, what should they read next?"

**Pros**:
- Most intelligent recommendations
- Understands nuanced context
- No training required

**Cons**:
- Expensive ($0.01-0.05 per recommendation)
- Latency (2-5 seconds)
- Unpredictable (hallucinations possible)

**Accuracy**: 80-90% (when context is good)

---

**RECOMMENDATION**: **Option C - Hybrid (Content + Collaborative)**

**Reasoning**:
- Start with content-based (immediate value)
- Add collaborative filtering as user data grows
- Reserve LLM for complex edge cases
- Cost-effective, performant, accurate
